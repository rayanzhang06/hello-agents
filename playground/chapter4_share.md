# 关于 Agent 经典范式，我能告诉你的六件事情

LangChain、AutoGen、CrewAI 这些框架可以让你很快跑起来一个 Agent。但当你需要处理"模型输出格式跑偏了怎么办""工具调用失败了怎么重试""Agent 在死循环里出不来怎么办"这些现实问题时，框架的抽象层反而成了障碍——你看不到问题出在哪一层，也不知道该在哪里下手。经典的 Agent 范式就是框架背后的设计蓝图。理解它们，你才能跳出"框架使用者"的身份，在标准组件无法满足需求时，有能力自己做架构决策。

你可能已经对 ReAct、Plan-and-Solve、Reflection 这些术语耳熟能详。但如果现在脱离框架和论文，让你用直觉回答：为什么这个场景必须用 ReAct 而不是 Plan-and-Solve？它们在“思考”与“行动”的耦合上到底有什么本质区别？如果你发现自己很难一针见血地讲清楚，那说明你还困在理解概念层面。

本文就试着给你一些我的理解。

假设你已经被科普过 ReAct、Plan-and-Solve、Reflection 的基本概念，希望以上内容能帮你真的理解它们。

---

## 1. 三种范式的区别

**ReAct**：边想边做，走一步看一步。

**Plan & Solve**：先想清楚再动手，一次性做完。

**Reflection**：做完之后再想一遍——做得好不好？能不能更好？

如果你只记住一件事：这三者的根本区别在于**思考和行动的耦合时机不同**。ReAct 把思考和行动交织在一起，P&S 把它们前后分离，而 Reflection 在"思考→行动"之上叠加了第二层思考——一种"对自己的输出进行审视"的元认知。

| 范式 | 思考何时发生 | 行动何时发生 | 两者关系 | 核心特征 |
|------|------------|------------|---------|---------|
| **ReAct** | 每步行动前 | 每次思考后 | 交织循环 | 局部最优，动态响应 |
| **Plan & Solve** | 任务最开始（一次性） | 规划完成后 | 先后分离 | 全局视图，静态执行 |
| **Reflection** | 两种思考：①生成时 + ②评审时 | 生成阶段 + 优化阶段 | 元认知循环 | 对输出结果进行批判性迭代 |

---

## 2. ReAct 擅长的和不擅长的

ReAct 可以描述为 “Thought → Action → Observation → Thought → ...”的循环。

它的核心能力是**动态响应**。搜索结果不对？下一轮换个关键词。工具调用失败？下一轮切换工具。这种"随时调整方向"的灵活性，让它天然适合探索式任务——查信息、回答实时性问题、与外部 API 交互。

但它有一个结构性盲区：**看不到全局**。每一步决策只能基于"历史观察 + 当下状态"，永远在问"下一步做什么"。

```markdown
举个例子：用户说“7点出门，帮我准备早晨”，涉及定闹钟、烧热水、煮咖啡、暖车。ReAct 会像个新手管家，做完一件再看下一件。但无法通过“以终为始”的倒推（如 6:00 热水、6:30 咖啡、6:50 暖车），确保所有任务在 7:00 准时就绪。

ReAct 缺乏这种全局编排能力，而这正是 Plan & Solve 的优势。

---

## 3. Plan & Solve 的全局感和它的代价

P&S 的工作方式是**先规划，再执行**。接到任务后，它会一次性生成完整的分步骤计划，然后逐步执行。

这意味着它拥有 ReAct 没有的全局视图——多步骤之间的顺序、依赖、甚至并行关系，都可以在规划阶段被考虑到。

但代价是：**计划是静态的**。

P&S 在规划阶段做出了所有决策，执行阶段只是"照做"。如果中途出现意外（比如预订机票时发现航班售空），纯 P&S 没有内置的重规划机制，它不会说"计划有变，让我重新想想"。

ReAct 在这里反而有优势。航班售空这种动态异常，ReAct 在下一个 Thought 循环中就能自然处理。

所以不是"P&S 比 ReAct 好"，而是它们各自看到了问题的不同面：**P&S 看全局，ReAct 看变化**。

---

## 4. Reflection 解决的是一个完全不同的问题

前两种范式关心的是**如何完成任务**。Reflection 关心的是**完成得好不好**。

这背后是一个思考层次的本质差异。ReAct 和 P&S 的思考都有一个共同特征：它们服务于"产生下一步行动"。而 Reflection 在此之上引入了第二层思考——**对行动结果的批判性审视**。这种"思考关于思考结果的思考"，就是元认知。

具体来说，它由两种性质不同的思考驱动：

- **执行器的思考**：为了产出答案而思考（主动生成）
- **审查器的思考**：对已有答案进行评判和反馈（元认知）

审查器并不直接产生行动，它产生的是**对行动质量的评估**。执行器根据这些评估优化答案，审查器再评估……循环往复，直到审查器认为"已经够好了"。

但要注意一个容易混淆的点：审查器评估的是**输出质量**，而无法感知**外部世界的变化**。它能评估不同算法的复杂度，并给出优化建议。但它不会自主搜索航班余票状态。

这意味着 Reflection 和 ReAct 处理的问题本质不同：
- ReAct 解决的是：**外部世界怎么变了？**
- Reflection 解决的是：**我的答案好不好？**

---

## 5. 成本：Reflection 的阿喀琉斯之踵？

Reflection 每迭代一轮至少额外调用 2 次 LLM（一次反思、一次优化），整个过程是串行的。三轮迭代就意味着至少 7 次 LLM 调用（1 次初始 + 3×2 次迭代），时延和费用都显著增长。

但这个问题是绝对的还是相对的？

举个例子：如果用户等着你的回复（聊天场景），3 轮迭代的时延确实难以接受。但如果场景允许异步——比如智能家居在睡前接收"帮我无痛早起"的指令，后台跑完 Reflection 优化方案，6 点前准备好——那时延可能不是问题。

始终综合考虑现实场景、产品设计、技术能力三者的适配和平衡，是智能体架构设计中非常重要的思维。

---

## 6. 真实世界需要的是组合架构

单一范式能覆盖的场景有限。在真实世界里，三种范式更适合作为积木，垒成一个分层架构。比如：

**最顶层：Reflection** → 质量保障。审查整体方案是否最优，甚至重新定义用户需求。

**中间层：Plan & Solve** → 编排与协调。接收高层指令，分解为子任务，搞定时序和优先级。

**执行层：ReAct** → 落地执行。调用具体工具，实时感知环境变化，处理动态异常。

回到"帮我无痛早起"这个例子：

- Reflection 作为审查层，评估方案是否真正符合"无痛"标准（依据用户历史习惯和健康建议）
- P&S 将目标分解为"6:00 渐进光照 → 6:20 调温 → 6:30 咖啡 → 6:50 暖车"
- 各设备上的 ReAct Agent 根据实时传感器数据执行任务

对于简单场景（"帮我开灯"），ReAct 单独就够了。但当任务的复杂度、时序要求和质量要求同时上升时，分层是自然而然的选择。
