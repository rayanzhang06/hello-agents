# 第四章 智能体经典范式 · 讨论笔记

> 本文档整理自第四章内容的学习讨论，结合习题分析，系统化三种经典 Agent 范式的本质、适用场景与组合策略。

---

## 一、三种范式的本质区别

### 1.1 从"思考与行动的组织方式"理解

| 范式 | 思考何时发生 | 行动何时发生 | 两者关系 | 核心特征 |
|------|------------|------------|---------|---------|
| **ReAct** | 每步行动前 | 每次思考后 | 交织循环 | 局部最优，动态响应 |
| **Plan & Solve** | 任务最开始（一次性） | 规划完成后 | 先后分离 | 全局视图，静态执行 |
| **Reflection** | 两种思考：①生成时 + ②评审时 | 生成阶段 + 优化阶段 | 元认知循环 | 对输出结果进行批判性迭代 |

**Reflection 的独特之处**：它不只是"思考→行动"，而是增加了"**思考关于行动结果的思考**"——元认知层次。这与另外两种范式的根本区别在于，它的第二种思考（审查器）并不直接产生行动，而是产生**对行动质量的评估与反馈**。

---

### 1.2 三种范式的决策视野对比

```
ReAct:    [当前状态] → 下一步
                          ↑
                    只看"现在"

P&S:      [问题] → [完整计划] → 步骤1 → 步骤2 → ... → 步骤n
                    一开始看"全局"，之后不再调整

Reflection: [行动] → [输出] → [审查] → [反馈] → [优化输出]
                                  ↑
                           持续评估"质量"
```

---

## 二、各范式的适用场景与局限

### 2.1 ReAct

**适用场景**：
- 需要实时响应外部变化的任务（如搜索信息、感知环境变化）
- 任务路径不可完全预知，需要"探索式"推进
- 对延迟敏感、需要及时反馈

**结构性局限**：
- **局部最优**：每次只考虑"下一步做什么"，缺乏全局视图
- **无法时序协调**：对于"6:00 热水 → 6:30 咖啡 → 6:50 暖车"这类需要提前统筹多步骤的任务，ReAct 无法在一开始就生成完整的时间表
- **不适合并行优化**：步进式决策会错过可以并行执行的机会

> 📌 **关键洞察**：ReAct 的"走一步看一步"在处理动态异常（如航班售空）时反而是优势——它可以在下一循环中自然转向替代方案。

---

### 2.2 Plan & Solve

**适用场景**：
- 任务可被清晰分解为有顺序依赖的子步骤
- 需要全局统筹（多步骤时序协调）
- 解答路径相对固定，不需要大量动态调整

**结构性局限**：
- **计划静态**：规划阶段一次性生成计划，执行阶段无法根据意外情况重新规划
- **无例外处理机制**：若某步骤失败（如机票售空），纯 P&S 没有内置的重规划能力

> 📌 **改进方向**：可在计划步骤中加入条件分支（"若 A 不可用，则执行 B"），但这增加了规划复杂度，且覆盖能力有限。

---

### 2.3 Reflection

**适用场景**：
- 对结果**质量**有高要求，"合格"不够，需要"优秀"
- 存在明确的评估维度（如算法效率、论文逻辑性、代码规范）
- 任务对时效性要求相对宽松（接受异步处理）
- 最优解可能需要**重新定义用户的需求本身**

**成本**：
- 每轮迭代至少额外调用 2 次 LLM（反思 + 优化），成本随迭代数线性增长
- 串行执行，总耗时显著延长

> 📌 **产品设计视角**：时延问题可通过产品设计缓解——提前告知用户"已接收任务，将在处理完成前通知你"，让 Reflection 在后台异步运行，用户无需等待。这将"绝对时延"变成了"感知时延"。

---

## 三、场景适用决策框架

```
任务是否需要外部信息/工具？
├─ 否 → 纯提示词工程或 Reflection（代码生成、文本优化等）
└─ 是 → 任务路径是否可提前确定？
         ├─ 是（结构化多步骤）→ Plan & Solve
         └─ 否（探索式/动态）→ ReAct

对结果质量有极高要求？
└─ 是 → 在以上基础上添加 Reflection 层
```

| 场景举例 | 推荐范式 | 理由 |
|---------|---------|------|
| 查询实时信息后回答问题 | ReAct | 探索式，依赖观察结果动态调整 |
| 多步骤数学推理 | P&S | 步骤明确，顺序固定 |
| 代码生成与优化 | Reflection | 质量导向，有明确的评估标准 |
| 智能家居单设备控制 | ReAct | 感知-行动闭环，实时响应 |
| 报告撰写（多信息源整合） | P&S | 先规划结构，再逐节填充 |
| 商务旅行预订（复杂约束） | P&S + ReAct | 全局规划 + 动态异常处理 |

## 四、组合使用：分层架构

单一范式很少能覆盖真实世界的复杂需求。三种范式可形成**分层架构**：

```
┌──────────────────────────────────────┐
│         Reflection（质量保障层）      │
│  审查整体方案是否最优，是否需要       │
│  重新定义用户需求                    │
├──────────────────────────────────────┤
│      Plan & Solve（编排层）          │
│  接收高层指令，分解为子任务，         │
│  协调时序与优先级                    │
├──────────────────────────────────────┤
│        ReAct（执行层）              │
│  执行具体子任务，实时感知环境变化，   │
│  处理动态异常                        │
└──────────────────────────────────────┘
```

**案例："帮我无痛早起"**

1. **Reflection**（质量层）：评估整体方案是否真正符合"无痛"标准，依据用户历史习惯和健康建议提供反馈
2. **P&S**（编排层）：将"无痛早起"分解为"6:00 渐进光照 → 6:20 调温 → 6:30 咖啡 → 6:50 暖车"等时序子任务
3. **ReAct**（执行层）：各设备 Agent 根据实时传感器数据执行任务（灯光、温控、咖啡机等）

**案例："预订北京→上海商务旅行"**

- 简单场景：P&S 规划行程框架 + ReAct 处理实时查询与异常（票价变动、售空）
- 复杂场景：外层加 Reflection，当无法在当前约束内找到最优解时，提出"建议用户调整会议时间以获得更优方案"

## 五、各范式实现关键要素

### 5.1 ReAct 实现要点

- **提示词格式**：必须约束 LLM 输出结构化格式（JSON 或 Thought/Action 文本），以便解析
- **最大步数限制（max_steps）**：防止无限循环的安全阀，建议 5-10 步
- **Observation 质量**：工具返回的观察结果质量直接影响下一步 Thought 的准确性
- **工具描述（Description）**：LLM 依赖 Description 选择工具，描述不清会导致工具误用

### 5.2 P&S 实现要点

- **规划输出格式**：强制约束为结构化格式（Python 列表）便于解析，避免自然语言带来的解析不稳定
- **执行器上下文**：每步执行时需传入：原始问题 + 完整计划 + 已完成步骤及结果，保证信息连贯
- **解析重试机制**：规划失败时（如格式不符）应有重试逻辑，建议最多 3 次后抛出异常

### 5.3 Reflection 实现要点

- **提示词特质化**：三个角色提示词风格需显著不同
  - 初始执行：资深工程师，专注功能实现
  - 反思审查：极其严格的评审专家，专注特定维度（如算法效率）
  - 优化改写：根据反馈重写，保持规范
- **停止条件**：除最大迭代数外，建议设置语义停止信号（如审查器返回"无需改进"）
- **Memory 模块**：保留完整历史轨迹（非仅最新版本），使执行器能感知自身的迭代路径
- **成本控制**：每轮至少 2 次额外 LLM 调用，生产环境需权衡质量与成本

## 六、延伸思考

### Reflection 的边界

Reflection 的审查器能评估**输出质量**，但无法感知**外部世界的实时状态**（如票是否售空）。因此：
- Reflection 解决"答案好不好"的问题
- ReAct 解决"外部世界怎么变了"的问题
- 两者处理的问题本质不同，不可相互替代

### 工具规模化的挑战

当可用工具数量扩展到 50-100 个时，将所有工具描述塞进 LLM 上下文会降低准确性。工程上的解决方向包括：
- 工具向量化 + 语义检索（Tool RAG）：先检索相关工具，再提供给 LLM
- 工具分层分组：按领域/功能预先聚类，LLM 先选分组再选工具
- Few-shot 工具调用示例：帮助 LLM 更准确地理解何时调用何种工具

### 用户习惯数据的角色

Reflection 审查器的反思质量取决于评估依据。当需要个性化评估（如"无痛"的标准因人而异）时，审查器需要接入用户历史数据——这指向了 **RAG（检索增强生成）** 和 **Memory 模块**的使用，也是第五章之后的核心话题。

根据以下描述解释 ReAct 范式运行机制：
第二幕：ReAct（边做边看，随机应变）

当你顺利完成第 1 步，正在执行第 2 步去山洞里“挖石头”时，突发状况来了！
- 看 (Observe)： 糟糕！一镐子敲下去，地下水突然涌了出来，要把你冲跑了！
- 想 (Reason)： 我不能再硬着头皮挖石头了，不然会被淹死或者冲进岩浆里。我得赶紧把出水口堵住。
- 做 (Act)： 马上切换到背包里的泥土块，对准漏水的地方“啪”地一下堵上去。
- 再看 (Observe)： 呼，水停了，安全了！继续安心挖石头。
(你看，ReAct 就是你在执行大计划的过程中，遇到突发的小麻烦时，帮你灵活应变的“急救包”。)
